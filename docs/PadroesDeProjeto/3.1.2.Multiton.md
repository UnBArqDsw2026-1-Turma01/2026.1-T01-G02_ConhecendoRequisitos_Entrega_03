## 1. Introdução

O **Multiton** é um padrão de projeto criacional que generaliza o padrão **Singleton**. Enquanto o Singleton garante que apenas **uma única instância** de uma classe exista no sistema, o Multiton permite um número **controlado e nomeado** de instâncias, indexadas por uma chave (GAMMA et al., 1995; SHALLOWAY; TROTT, 2004).

A estrutura central do Multiton é um mapa (dicionário/objeto) estático interno à própria classe, que armazena as instâncias já criadas. Quando uma instância é solicitada:

1. Se a chave **já existe** no mapa → a instância existente é retornada.
2. Se a chave **não existe** → uma nova instância é criada, armazenada e retornada.

Esse comportamento garante que para cada chave haja exatamente **uma instância**, sem mais e sem menos.

### Quando aplicar?

- Quando o sistema precisa de **múltiplos objetos únicos**, identificados por categorias ou chaves bem definidas;
- Quando se deseja evitar a criação redundante de objetos que representam o mesmo "contexto" (ex.: conexão com banco de dados por schema, perfil de usuário por tipo, configurações por módulo);
- Quando a quantidade de instâncias é **finita e previsível**, e cada uma delas precisa ser compartilhada globalmente dentro de seu contexto.

## 2. Metodologia

A plataforma **ConhecendoRequisitos** possui um banco de dados relacional com **13 tabelas**, organizadas em 4 grupos semânticos distintos, conforme definido no [Diagrama Lógico de Dados (DLD)](https://unbarqdsw2026-1-turma01.github.io/2026.1-T01-_G2_ConhecendoRequisitos_Entrega_02/#/Modelagem/2.5.5.DiagramaLogico) da [Entrega 02](https://unbarqdsw2026-1-turma01.github.io/2026.1-T01-_G2_ConhecendoRequisitos_Entrega_02/#/). Cada grupo possui afinidade funcional e é acessado por módulos diferentes da aplicação:

| Contexto (chave) | Tabelas                                                          | Módulo da aplicação                              |
| ---------------- | ---------------------------------------------------------------- | ------------------------------------------------ |
| `"usuarios"`     | `Usuario`, `Aluno`, `Administrador`                              | Autenticação, perfis (US01–US07)                 |
| `"conteudo"`     | `Trilha`, `Modulo`, `Conteudo`, `Quiz`, `Questao`, `Alternativa` | Gestão de conteúdo (US08–US11, US13, US15, US17) |
| `"progresso"`    | `ProgressoTrilha`, `ProgressoModulo`, `ProgressoConteudo`        | Rastreamento de progresso (US14, US16, US18)     |
| `"quizzes"`      | `TentativaQuiz`                                                  | Tentativas de quiz (US19–US24)                   |

Sem o Multiton, cada módulo que precisasse acessar o banco abriria sua própria conexão — resultando em conexões duplicadas para o mesmo grupo de tabelas, desperdício de recursos e dificuldade de controle. Com o Multiton aplicado à classe **`ConexaoDB`**, garante-se que:

- Toda parte do sistema que solicitar `ConexaoDB.getConexao("usuarios")` receberá **sempre a mesma conexão**;
- O mesmo vale para cada um dos demais contextos.

### Relação com o Singleton

O Multiton é frequentemente descrito como um "Singleton indexado por chave". Enquanto o Singleton é o caso especial com exatamente uma chave, o Multiton generaliza o conceito. No contexto do projeto, o Singleton seria insuficiente pois há **quatro contextos distintos e simultâneos** que precisam coexistir, cada um com seu ciclo de vida e conjunto de tabelas próprios.

## 4. Descrição das Classes

### 4.1. `ConexaoDB` (Multiton)

Classe central do padrão. Mantém internamente um `Map` estático privado `#pool` que indexa cada conexão pelo nome do contexto.

| Atributo / Método                 | Visibilidade     | Descrição                                                                     |
| --------------------------------- | ---------------- | ----------------------------------------------------------------------------- |
| `#pool: Map<string, ConexaoDB>`   | `private static` | Mapa que armazena as conexões abertas, indexadas pelo contexto                |
| `#contexto: string`               | `private`        | Nome do contexto desta conexão (`"usuarios"`, `"conteudo"`, etc.)             |
| `#tabelas: string[]`              | `private`        | Lista de tabelas do DLD que esta conexão gerencia                             |
| `#descricao: string`              | `private`        | Descrição semântica do contexto                                               |
| `#status: string`                 | `private`        | Estado da conexão: `"conectando"` → `"conectado"` → `"fechado"`               |
| `#totalQueries: number`           | `private`        | Contador de queries executadas nesta conexão                                  |
| `#ultimaQuery: string`            | `private`        | Timestamp da última query executada                                           |
| `constructor(contexto)`           | `private`        | Construtor privado — impede instanciação direta pelo cliente                  |
| `#conectar()`                     | `private`        | Abre a conexão com o banco (simulada; em produção, chamaria o driver do SGBD) |
| `getConexao(contexto): ConexaoDB` | `public static`  | **Ponto de acesso global.** Retorna conexão existente ou cria uma nova        |
| `query(sql): object`              | `public`         | Executa uma query SQL no contexto desta conexão                               |
| `getContexto(): string`           | `public`         | Retorna o nome do contexto                                                    |
| `getTabelas(): string[]`          | `public`         | Retorna cópia defensiva das tabelas gerenciadas                               |
| `getStatus(): string`             | `public`         | Retorna o status da conexão                                                   |
| `exibirStatus()`                  | `public`         | Exibe estado completo da conexão no console                                   |
| `quantidadeDeConexoes(): number`  | `public static`  | Retorna quantas conexões estão abertas no pool                                |
| `listarConexoes(): string[]`      | `public static`  | Lista os contextos com conexão aberta                                         |
| `resetar()`                       | `public static`  | Fecha todas as conexões e limpa o pool                                        |

## 5. Implementação

O código a seguir implementa o padrão Multiton em **JavaScript (Node.js)**, aplicado ao gerenciamento de pool de conexões com banco de dados na plataforma ConhecendoRequisitos. A implementação está dividida em configuração, classe e demonstração.

### 5.1. `ConexaoDB.js` (O Multiton)

<details>
<summary><b>Ver Código Fonte</b></summary>

```javascript
const CONTEXTOS_DB = require("./config/contextos");

class ConexaoDB {
  // Mapa estático privado — núcleo do Multiton
  static #pool = new Map();

  #contexto;
  #tabelas;
  #status;
  #totalQueries;

  constructor(contexto) {
    if (!CONTEXTOS_DB[contexto]) {
      throw new Error(`Contexto de banco inválido: "${contexto}".`);
    }

    const config = CONTEXTOS_DB[contexto];
    this.#contexto = contexto;
    this.#tabelas = config.tabelas;
    this.#status = "conectando";
    this.#totalQueries = 0;

    this.#conectar();
  }

  #conectar() {
    this.#status = "conectado";
    console.log(
      `[ConexaoDB] ✓ Conexão aberta para contexto "${this.#contexto}"`,
    );
  }

  // Ponto de acesso global
  static getConexao(contexto) {
    if (!ConexaoDB.#pool.has(contexto)) {
      ConexaoDB.#pool.set(contexto, new ConexaoDB(contexto));
    } else {
      console.log(
        `[ConexaoDB] ↩ Conexão reutilizada para contexto "${contexto}"`,
      );
    }
    return ConexaoDB.#pool.get(contexto);
  }

  query(sql) {
    if (this.#status !== "conectado") throw new Error("Conexão inativa");
    this.#totalQueries += 1;
    console.log(
      `[${this.#contexto.toUpperCase()}] Query executada: "${sql.trim()}"`,
    );
  }

  static resetar() {
    ConexaoDB.#pool.clear();
  }
}

module.exports = ConexaoDB;
```

</details>

### 5.2. `index.js` (Demonstração de Uso)

<details>
<summary><b>Ver Código Fonte</b></summary>

```javascript
const ConexaoDB = require("./ConexaoDB");

console.log("--- Abrindo conexão para 'usuarios' (1ª vez) ---");
const dbUsuarios1 = ConexaoDB.getConexao("usuarios");

console.log("\n--- Solicitando 'usuarios' novamente (2ª vez) ---");
const dbUsuarios2 = ConexaoDB.getConexao("usuarios");

console.log("\n--- Abrindo conexão para 'conteudo' ---");
const dbConteudo = ConexaoDB.getConexao("conteudo");

console.log("\n── Verificação de identidade ─────────────────────────");
console.log(
  `dbUsuarios1 === dbUsuarios2 (mesma conexão?): ${dbUsuarios1 === dbUsuarios2}`,
);
console.log(
  `dbUsuarios1 === dbConteudo  (contextos diferentes?): ${dbUsuarios1 === dbConteudo}`,
);

console.log("\n── Executando queries por contexto ───────────────────");
dbUsuarios1.query("SELECT * FROM Usuario WHERE tipo = 'aluno'");
dbConteudo.query("SELECT * FROM Trilha ORDER BY ordem");
```

</details>

### 5.3. Saída Esperada

```
--- Abrindo conexão para 'usuarios' (1ª vez) ---
[ConexaoDB] ✓ Conexão aberta para contexto "usuarios"

--- Solicitando 'usuarios' novamente (2ª vez) ---
[ConexaoDB] ↩ Conexão reutilizada para contexto "usuarios"

--- Abrindo conexão para 'conteudo' ---
[ConexaoDB] ✓ Conexão aberta para contexto "conteudo"

── Verificação de identidade ─────────────────────────
dbUsuarios1 === dbUsuarios2 (mesma conexão?): true
dbUsuarios1 === dbConteudo  (contextos diferentes?): false

── Executando queries por contexto ───────────────────
[USUARIOS] Query executada: "SELECT * FROM Usuario WHERE tipo = 'aluno'"
[CONTEUDO] Query executada: "SELECT * FROM Trilha ORDER BY ordem"
```

> **Ponto-chave**: Observe que `dbUsuarios1 === dbUsuarios2` é `true`. Isso prova que **não foi aberta uma nova conexão** na segunda chamada — a conexão do pool foi reutilizada, ilustrando a essência do Multiton.

## 7. Senso Crítico

Quando começamos a explorar o padrão GOF Multiton, a ideia de gerenciar instâncias a partir de chaves nos pareceu, inicialmente, um pouco complexa de encaixar no escopo funcional do "ConhecendoRequisitos". Tentávamos adaptar o padrão de diversas formas, chegando até a projetá-lo para controlar sessões de usuários logados, mas não conseguíamos visualizar onde ele realmente agregaria valor estrutural a longo prazo. Então, após olharmos os artefatos realizados durante as entregas anteriores, e encontrarmos o [Diagrama Lógico de Dados](https://unbarqdsw2026-1-turma01.github.io/2026.1-T01-_G2_ConhecendoRequisitos_Entrega_02/#/Modelagem/2.5.5.DiagramaLogico) realizado na [Entrega 2](https://unbarqdsw2026-1-turma01.github.io/2026.1-T01-_G2_ConhecendoRequisitos_Entrega_02/#/), percebemos que o padrão Multiton poderia ser utilizado para gerenciar conexões com o banco de dados.

Ao aplicarmos a abordagem do Multiton para atuar como um _pool_ de conexões de banco de dados (`ConexaoDB`), vimos na prática como ele resolve um problema real. Diferente do Singleton tradicional, que nos prenderia a uma única conexão global (o que poderia gerar gargalos de transação e lentidão no sistema inteiro), o Multiton nos permitiu instanciar uma conexão dedicada para cada contexto semântico do banco (usuários, conteúdos, progresso e quizzes). Essa arquitetura moderna promove um isolamento de falhas excelente: se houver uma sobrecarga no sistema de quizzes, o módulo de autenticação de usuários continuará operando normalmente.

O Singleton não era a solução mais escalável para a nossa infraestrutura de dados, mas o Multiton provou ser uma excelente solução para gerenciar múltiplas instâncias controladas.

A implementação deste padrão gerou debates riquíssimos na equipe. Foram horas de debate no Whatsapp e Teams inteiramente dedicadas a entender os mecanismos internos do Multiton e como a linguagem JavaScript lida com instâncias e atributos estáticos. O padrão foi de difícil compreensão e aplicação pois exigia um conhecimento mais aprofundado da orientação a objetos.

Este modelo valida a evolução da modelagem para a implementação física do sistema, fornecendo um mecanismo extensível para lidar com a demanda de usuários de forma modular.
