## Introdução

O presente documento tem como objetivo apresentar a implementação do padrão de projeto Singleton aplicado à classe `RepositorioDeConteudo`, desenvolvida no contexto do nosso projeto acadêmico.
O padrão Singleton é um dos padrões criacionais do catálogo GoF (Gang of Four) e tem como finalidade garantir que uma classe possua apenas uma instância em todo o sistema, fornecendo um ponto global de acesso a ela (Refactoring Guru, 2025).

## Motivação

A motivação principal para o uso do Singleton na classe `RepositorioDeConteudo` é garantir o controle centralizado do catálogo de trilhas, módulos e conteúdos em um único ponto do sistema. Com isso, evita-se a criação de múltiplas instâncias que poderiam ocasionar inconsistências de dados, como estruturas de conteúdo divergentes entre diferentes telas ou módulos da aplicação. Assim, o Administrador que cria uma trilha, o Aluno que a consulta, e a tela de visualização de progresso, todos acessam a mesma base de conteúdo.

## Desenvolvimento

A implementação foi feita em JavaScript seguindo a mesma linha dos demais exemplos criacionais do projeto. A classe `PublicacoesStore` mantém o estado das publicações em memória e expõe um ponto de acesso global por meio do método estático `getInstance()`. Assim, toda vez que o sistema pede a store, recebe exatamente o mesmo objeto, com a mesma lista de dados e os mesmos assinantes registrados.

Na prática, a classe funciona como uma store centralizada para a interface. Ela oferece operações para adicionar, atualizar, remover, listar e limpar publicações. Além disso, possui um mecanismo de assinatura para notificar componentes ou consumidores sempre que o estado muda. Esse detalhe é importante porque facilita o uso com React: a interface pode observar a store e reagir automaticamente às alterações sem duplicar estado local desnecessário.

O comportamento do Singleton foi reforçado com três pontos principais:

1. O construtor impede a instanciação direta fora do método `getInstance()`.
2. A instância é armazenada em um atributo estático privado.
3. As operações de escrita sempre atuam sobre o mesmo conjunto de publicações.

Com isso, o projeto passa a ter um único ponto de verdade para o gerenciamento das publicações, o que reduz inconsistências e simplifica a manutenção.

## Metodologia

No contexto da plataforma **ConhecendoRequisitos**, o repositório de conteúdo precisa ser único e compartilhado por todos os usuários (Alunos e Administradores). O Singleton foi aplicado para gerenciar centralmente três entidades de domínio:

- **Trilha**: representa um caminho de aprendizagem completo, com identificador, título, descrição e ordem;
- **Modulo**: organiza os blocos intermediários de conteúdo dentro de uma trilha, com referência ao `idTrilha`;
- **Conteudo**: representa o material final consumido pelo aluno (vídeos, textos, exercícios, etc.), com referência ao `idModulo`.

Sem o padrão Singleton, múltiplas instâncias do repositório poderiam causar divergência entre a visão do Administrador (que cria conteúdo) e a do Aluno (que o consome), além de inconsistências em componentes de interface que visualizam progresso simultaneamente. Com o Singleton, existe um ponto único de verdade: qualquer alteração feita em uma referência é imediatamente visível em todas as outras, e componentes podem se inscrever em mudanças para atualizar a interface automaticamente.

### Descrição das Classes

#### `RepositorioDeConteudo` (Singleton)

Mantém a instância única do repositório e centraliza todas as operações sobre trilhas, módulos e conteúdos. Implementa um mecanismo de assinatura para notificar observadores quando o estado muda.

| Atributo / Método                                            | Descrição                                               |
| ------------------------------------------------------------ | ------------------------------------------------------- |
| `#instance: static`                                          | Armazena a instância única (privada e estática)         |
| `#trilhas, #modulos, #conteudos: Array`                      | Vetores que armazenam as entidades                      |
| `#assinantes: Set`                                           | Conjunto de listeners que observam mudanças             |
| `getInstance(): RepositorioDeConteudo`                       | Retorna a instância única, criando-a se necessário      |
| `adicionarTrilha(titulo, descricao, ordem): Trilha`          | Cria nova trilha e notifica assinantes                  |
| `listarTrilhas(): Trilha[]`                                  | Retorna cópia de todas as trilhas                       |
| `atualizarTrilha(id, dados): Trilha`                         | Atualiza trilha e notifica assinantes                   |
| `removerTrilha(id): boolean`                                 | Remove trilha e conteúdo associado                      |
| `adicionarModulo(idTrilha, titulo, descricao): Modulo`       | Cria novo módulo                                        |
| `listarModulosPorTrilha(idTrilha): Modulo[]`                 | Lista módulos de uma trilha                             |
| `adicionarConteudo(idModulo, titulo, tipo, dados): Conteudo` | Cria novo conteúdo                                      |
| `listarConteudosPorModulo(idModulo): Conteudo[]`             | Lista conteúdos de um módulo                            |
| `inscrever(listener): Function`                              | Registra um observer e retorna função para desinscrever |

#### `Trilha`, `Modulo`, `Conteudo` (Domain)

Classes que representam as entidades de domínio do repositório. Cada uma possui um identificador único e relacionamentos apropriados.

## Implementação

O código a seguir implementa o padrão Singleton em **JavaScript (Node.js)**, seguindo a mesma abordagem dos demais padrões criacionais do projeto. A implementação garante que apenas uma instância de `RepositorioDeConteudo` exista durante todo o ciclo de vida da aplicação, enquanto oferece uma API clara para operações CRUD sobre as entidades de conteúdo.

```javascript
// ─────────────────────────────────────────
// DOMÍNIO: Conteúdo educacional
// ─────────────────────────────────────────

/**
 * Representa uma trilha de aprendizagem.
 */
class Trilha {
  constructor(id, titulo, descricao, ordem) {
    this.id = id;
    this.titulo = titulo;
    this.descricao = descricao;
    this.ordem = ordem;
  }
}

/**
 * Representa um módulo dentro de uma trilha.
 */
class Modulo {
  constructor(id, idTrilha, titulo, descricao) {
    this.id = id;
    this.idTrilha = idTrilha;
    this.titulo = titulo;
    this.descricao = descricao;
  }
}

/**
 * Representa um conteúdo (vídeo, texto, exercício, etc) dentro de um módulo.
 */
class Conteudo {
  constructor(id, idModulo, titulo, tipo, dados) {
    this.id = id;
    this.idModulo = idModulo;
    this.titulo = titulo;
    this.tipo = tipo; // 'video', 'texto', 'exercicio', etc
    this.dados = dados;
  }
}

// ─────────────────────────────────────────
// SINGLETON: Repositório centralizado
// ─────────────────────────────────────────

/**
 * Classe Singleton que centraliza o gerenciamento de trilhas,
 * módulos e conteúdos educacionais.
 */
class RepositorioDeConteudo {
  static #instance = null;

  #trilhas;
  #modulos;
  #conteudos;
  #proximoIdTrilha;
  #proximoIdModulo;
  #proximoIdConteudo;
  #assinantes;

  constructor() {
    // Impede instanciação fora de getInstance()
    if (RepositorioDeConteudo.#instance) {
      throw new Error(
        "RepositorioDeConteudo é um Singleton. Use RepositorioDeConteudo.getInstance().",
      );
    }

    this.#trilhas = [];
    this.#modulos = [];
    this.#conteudos = [];
    this.#proximoIdTrilha = 1;
    this.#proximoIdModulo = 1;
    this.#proximoIdConteudo = 1;
    this.#assinantes = new Set();
  }

  /**
   * Retorna a instância única do repositório.
   * Cria-a na primeira chamada.
   */
  static getInstance() {
    if (!RepositorioDeConteudo.#instance) {
      RepositorioDeConteudo.#instance = new RepositorioDeConteudo();
    }
    return RepositorioDeConteudo.#instance;
  }

  // ────── Operações com Trilhas ──────

  adicionarTrilha(titulo, descricao, ordem) {
    const trilha = new Trilha(this.#proximoIdTrilha, titulo, descricao, ordem);
    this.#proximoIdTrilha += 1;
    this.#trilhas.push(trilha);
    this.#notificar();
    return { ...trilha };
  }

  obterTrilhaPorId(id) {
    const trilha = this.#trilhas.find((t) => t.id === id);
    return trilha ? { ...trilha } : null;
  }

  listarTrilhas() {
    return this.#trilhas.map((t) => ({ ...t }));
  }

  atualizarTrilha(id, dados) {
    const trilha = this.#trilhas.find((t) => t.id === id);
    if (!trilha) return null;

    Object.assign(trilha, dados);
    this.#notificar();
    return { ...trilha };
  }

  removerTrilha(id) {
    const indice = this.#trilhas.findIndex((t) => t.id === id);
    if (indice === -1) return false;

    this.#trilhas.splice(indice, 1);
    // Remove módulos e conteúdos relacionados
    this.#modulos = this.#modulos.filter((m) => m.idTrilha !== id);
    this.#conteudos = this.#conteudos.filter(
      (c) => !this.#modulos.some((m) => m.id === c.idModulo),
    );
    this.#notificar();
    return true;
  }

  // ────── Operações com Módulos ──────

  adicionarModulo(idTrilha, titulo, descricao) {
    const modulo = new Modulo(
      this.#proximoIdModulo,
      idTrilha,
      titulo,
      descricao,
    );
    this.#proximoIdModulo += 1;
    this.#modulos.push(modulo);
    this.#notificar();
    return { ...modulo };
  }

  obterModuloPorId(id) {
    const modulo = this.#modulos.find((m) => m.id === id);
    return modulo ? { ...modulo } : null;
  }

  listarModulosPorTrilha(idTrilha) {
    return this.#modulos
      .filter((m) => m.idTrilha === idTrilha)
      .map((m) => ({ ...m }));
  }

  atualizarModulo(id, dados) {
    const modulo = this.#modulos.find((m) => m.id === id);
    if (!modulo) return null;

    Object.assign(modulo, dados);
    this.#notificar();
    return { ...modulo };
  }

  removerModulo(id) {
    const indice = this.#modulos.findIndex((m) => m.id === id);
    if (indice === -1) return false;

    this.#modulos.splice(indice, 1);
    this.#conteudos = this.#conteudos.filter((c) => c.idModulo !== id);
    this.#notificar();
    return true;
  }

  // ────── Operações com Conteúdos ──────

  adicionarConteudo(idModulo, titulo, tipo, dados) {
    const conteudo = new Conteudo(
      this.#proximoIdConteudo,
      idModulo,
      titulo,
      tipo,
      dados,
    );
    this.#proximoIdConteudo += 1;
    this.#conteudos.push(conteudo);
    this.#notificar();
    return { ...conteudo };
  }

  obterConteudoPorId(id) {
    const conteudo = this.#conteudos.find((c) => c.id === id);
    return conteudo ? { ...conteudo } : null;
  }

  listarConteudosPorModulo(idModulo) {
    return this.#conteudos
      .filter((c) => c.idModulo === idModulo)
      .map((c) => ({ ...c }));
  }

  atualizarConteudo(id, dados) {
    const conteudo = this.#conteudos.find((c) => c.id === id);
    if (!conteudo) return null;

    Object.assign(conteudo, dados);
    this.#notificar();
    return { ...conteudo };
  }

  removerConteudo(id) {
    const indice = this.#conteudos.findIndex((c) => c.id === id);
    if (indice === -1) return false;

    this.#conteudos.splice(indice, 1);
    this.#notificar();
    return true;
  }

  // ────── Sistema de Observadores ──────

  /**
   * Registra um listener que será chamado sempre que o repositório muda.
   * Retorna uma função para desinscrever.
   */
  inscrever(listener) {
    if (typeof listener !== "function") {
      return () => {};
    }

    this.#assinantes.add(listener);

    return () => {
      this.#assinantes.delete(listener);
    };
  }

  /**
   * Notifica todos os listeners sobre mudanças no repositório.
   */
  #notificar() {
    const snapshot = {
      trilhas: this.listarTrilhas(),
      modulos: this.#modulos.map((m) => ({ ...m })),
      conteudos: this.#conteudos.map((c) => ({ ...c })),
    };

    this.#assinantes.forEach((listener) => {
      try {
        listener(snapshot);
      } catch (erro) {
        console.error("Erro ao notificar listener:", erro);
      }
    });
  }
}

module.exports = RepositorioDeConteudo;
```

## Funcionalidade

O funcionamento do Singleton pode ser observado em três níveis. No primeiro, `RepositorioDeConteudo.getInstance()` cria o objeto apenas uma vez no ciclo de vida da aplicação. No segundo, os métodos de manipulação de trilhas, módulos e conteúdos operam sobre os mesmos vetores internos, independentemente de qual referência acione a classe. No terceiro, o mecanismo de assinatura permite que componentes se inscrevam e recebam notificações sempre que o catálogo muda.

Em uma aplicação React, isso permite que diferentes telas consultem o mesmo catálogo sem criar cópias independentes. Se um Administrador criar uma nova trilha, todos os Alunos e componentes de interface que observam o repositório veem imediatamente a mudança, mantendo a aplicação coerente com uma única fonte de verdade.

## Senso crítico

A escolha do Singleton para `RepositorioDeConteudo` é apropriada porque existe um catálogo de conteúdo educacional que deve ser único e compartilhado por toda a aplicação. Manter instâncias separadas do repositório não traria nenhum benefício — pelo contrário, aumentaria o risco de: divergência de dados (uma trilha criada não aparecer em todas as telas), perda de sincronização durante atualizações, e inconsistência de estado.

Com o padrão, o repositório oferece uma API clara com operações organizadas por entidade (trilhas, módulos, conteúdos) e um sistema de assinatura que mantém a interface reativa às mudanças. Por outro lado, o Singleton também exige prudência: quando usado em excesso, pode criar dependência global excessiva e dificultar testes. Neste projeto, o risco é mitigado porque o repositório tem responsabilidade bem definida (gerenciar conteúdo educacional) e uma API pequena, sem exposição excessiva de estado interno.

Além disso, a integração com React mostrou alguns benfícios (centralização e a previsibilidade do estado) mas também uma cautela em seu uso, especialmente em relação ao acoplamento global e à testabilidade.

## Conclusão

A aplicação do Singleton na classe `RepositorioDeConteudo` demonstra como um ponto único de acesso pode eliminar inconsistências e simplificar o compartilhamento de um catálogo de conteúdo entre diferentes partes da aplicação. A solução atende bem ao cenário proposto, permitindo que Administradores criem conteúdo, Alunos o consumam e componentes de visualização de progresso operem sempre sobre o mesmo estado, sem duplicação ou divergência.

Combinada com React via hook `useRepositorioDeConteudo`, a abordagem oferece reatividade sem complexidade adicional. O exemplo também reforça um aprendizado importante: o Singleton deve ser reservado para casos em que o estado global é realmente necessário. Quando aplicado com critério e responsabilidade bem definida, ele contribui para uma arquitetura mais organizada, previsível e fácil de manter no contexto de um projeto educacional como o ConhecendoRequisitos.
