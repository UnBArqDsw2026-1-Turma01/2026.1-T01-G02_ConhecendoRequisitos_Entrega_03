## 1. Introdução

O **Proxy** é um padrão de projeto estrutural catalogado pela Gang of Four (GoF) que fornece um **substituto ou representante** de outro objeto para controlar o acesso a ele (GAMMA et al., 1995). O Proxy mantém a mesma interface do objeto real, permitindo que o cliente interaja de forma transparente, sem saber se está acessando o Proxy ou o serviço real.

Existem diferentes tipos de Proxy, como o _Virtual Proxy_ (que adia a criação de objetos pesados), o _Remote Proxy_ (representa um objeto remoto) e o _Protection Proxy_ (controla acesso baseado em permissões). Neste projeto, aplicamos o **Protection Proxy**.

### Quando aplicar?

- Quando é necessário **controlar o acesso** a um objeto, garantindo que apenas usuários autorizados executem determinadas operações;
- Quando deseja-se adicionar verificações de segurança ou lógicas de controle de permissão sem modificar o código do serviço real (respeitando o Princípio Aberto/Fechado);
- Quando as regras de acesso dependem do perfil ou estado do cliente.

## 2. Metodologia

No contexto da plataforma **ConhecendoRequisitos**, existem diferentes perfis de usuário, como **Aluno** e **Administrador**. Muitas operações de gerenciamento de conteúdo (criar trilhas, módulos, quizzes) e de plataforma (visualizar estatísticas globais) devem ser restritas aos administradores. Além disso, operações de gerenciamento de perfil devem ser limitadas para que um aluno só possa editar, excluir ou visualizar os detalhes de seu próprio perfil, enquanto administradores podem gerenciar o perfil de qualquer usuário.

Sem o padrão Proxy, essas verificações de permissão teriam que ser espalhadas diretamente dentro dos métodos do serviço real, violando o Princípio da Responsabilidade Única (SRP) e dificultando a manutenção. Com a implementação de um **Protection Proxy**, isolamos toda a lógica de validação de acesso na classe proxy. O serviço real foca apenas na regra de negócio, enquanto o proxy atua como um "porteiro", filtrando as requisições baseadas no tipo e identidade do usuário.

### 4.2. `UsuarioServiceReal` (RealSubject)

Herda de `UsuarioService` e implementa as funcionalidades reais.

| Atributo / Método               | Descrição                                                              |
| ------------------------------- | ---------------------------------------------------------------------- |
| `trilhas`, `modulos`, `quizzes` | Listas simulando o armazenamento de dados do sistema                   |
| _Implementação dos 13 métodos_  | Realizam efetivamente a criação, edição, exclusão e exibição dos dados |

### 4.3. `UsuarioServiceProxy` (Proxy)

Herda de `UsuarioService` e intercepta as chamadas.

| Atributo / Método              | Descrição                                                                                                  |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------- |
| `serviceReal: UsuarioService`  | Referência ao serviço real para o qual delegará as chamadas                                                |
| `usuario: Usuario`             | O usuário atual autenticado, cujo papel será avaliado                                                      |
| `_verificarPermissaoAdmin()`   | Helper interno que checa se o usuário é "administrador"                                                    |
| `_verificarAcessoPerfil()`     | Helper interno que checa se o usuário é administrador ou o dono do perfil acessado                         |
| _Implementação dos 13 métodos_ | Interceptam a requisição, executam as checagens e, se autorizados, repassam a chamada para o `serviceReal` |

// UsuarioService.js — Interface (Subject)
class UsuarioService {
  constructor() {
    if (new.target === UsuarioService) {
      throw new Error("UsuarioService é uma interface abstrata.");
    }
  }
  // Declaração dos 13 métodos abstratos...
}

// UsuarioServiceReal.js — RealSubject
class UsuarioServiceReal extends UsuarioService {
  constructor() {
    super();
    this.trilhas = [];
    this.modulos = [];
    this.quizzes = [];
  }

  criarTrilha(titulo, descricao) {
    const id = this.trilhas.length + 1;
    this.trilhas.push({ id, titulo, descricao });
    console.log(`  ✔ Trilha '${titulo}' criada com id=${id}.`);
  }
  // Implementação dos demais métodos reais...
}

// UsuarioServiceProxy.js — Proxy (Protection Proxy)
class UsuarioServiceProxy extends UsuarioService {
  constructor(serviceReal, usuario) {
    super();
    this.serviceReal = serviceReal;
    this.usuario = usuario;
  }

  _verificarPermissaoAdmin(operacao) {
    if (!this.usuario.ehAdministrador()) {
      console.log(
        `  ✘ ACESSO NEGADO: '${this.usuario.nome}' não tem permissão para ${operacao}.`,
      );
      return false;
    }
    return true;
  }

  _verificarAcessoPerfil(email, operacao) {
    if (this.usuario.ehAdministrador()) return true;
    if (this.usuario.email !== email) {
      console.log(
        `  ✘ ACESSO NEGADO: '${this.usuario.nome}' não pode ${operacao} perfil de '${email}'.`,
      );
      return false;
    }
    return true;
  }

  criarTrilha(titulo, descricao) {
    if (!this._verificarPermissaoAdmin("criar trilha")) return;
    this.serviceReal.criarTrilha(titulo, descricao);
  }

  editarUsuario(email, nome, senha) {
    if (!this._verificarAcessoPerfil(email, "editar o")) return;
    this.serviceReal.editarUsuario(email, nome, senha);
  }
  // Interceptação dos demais métodos...
}

## 6. Senso Crítico

A aplicação do padrão Proxy, especificamente na vertente de **Protection Proxy**, trouxe ganhos imensos para o controle de acesso e para a manutenibilidade do sistema. Ao isolar a verificação de permissões do núcleo de negócio, aderimos de forma exemplar aos princípios SOLID.

Primeiramente, o _Single Responsibility Principle_ (SRP) é favorecido: o `UsuarioServiceReal` só se preocupa com a lógica de gerenciar conteúdos e usuários, deixando a cargo do `UsuarioServiceProxy` toda a lógica de segurança e controle de acesso. Caso seja necessário alterar a política de segurança, modificamos apenas o Proxy; se a lógica de criação de trilhas mudar, alteramos o RealSubject.

Além disso, conseguimos respeitar o _Open/Closed Principle_ (OCP). Se amanhã a plataforma decidir implementar logs de todas as ações de usuários (um _Logging Proxy_), ou realizar o carregamento lento de perfis de usuário a partir do banco de dados (_Virtual Proxy_), novos Proxies podem ser encapsulados ou encadeados sem alterar a interface `UsuarioService` ou a implementação de `UsuarioServiceReal`.

Como ponto de reflexão, o Proxy neste cenário pode apresentar redundância caso o sistema cresça e utilize frameworks ou middlewares especializados em autorização (como `Auth Guards` ou anotações). Porém, a nível arquitetural e didático, representa perfeitamente o comportamento de bloqueio condicional de acesso, mitigando falhas de segurança por requisições não autorizadas.

## 9. Conclusão

A utilização do padrão de projeto estrutural **Proxy** se mostrou essencial e eficaz para garantir o controle de acesso seguro na plataforma "ConhecendoRequisitos". Através do Protection Proxy, blindamos as operações administrativas exclusivas e o gerenciamento de perfis, mantendo o código do serviço real limpo e livre de lógica de permissões. O resultado foi um sistema robusto contra acessos indevidos e estruturalmente preparado para expansões futuras com baixíssimo acoplamento.
