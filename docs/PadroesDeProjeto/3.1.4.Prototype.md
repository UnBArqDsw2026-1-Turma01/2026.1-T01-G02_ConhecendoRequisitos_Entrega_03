## 1. Introdução

O **Prototype** é um padrão de projeto criacional que permite copiar objetos existentes sem fazer o seu código ficar dependente das suas classes. Ele delega o processo de clonagem para os próprios objetos que estão sendo clonados.

### Quando aplicar?

- Quando a criação de um objeto é custosa ou complexa (por exemplo, requer muitas consultas ao banco de dados ou inicializações demoradas);
- Quando o sistema precisa criar cópias de objetos pré-configurados com estados específicos;
- Para reduzir a proliferação de subclasses que diferem apenas em sua configuração inicial.

---

## 2. Metodologia

Na plataforma **ConhecendoRequisitos**, a estruturação didática é composta por uma hierarquia profunda: uma **Trilha** possui vários **Módulos**, que contêm **Quizzes**, que são compostos por **Questões**, as quais possuem **Alternativas**. 

Quando um administrador deseja criar uma nova versão de uma Trilha, ou reaproveitar um Quizz padrão para diferentes módulos, criar toda essa estrutura do zero seria extremamente custoso e repetitivo. Aplicando o padrão **Prototype**, podemos simplesmente clonar uma `Trilha` inteira ou um `Quizz` (incluindo todas as suas dependências aninhadas) e apenas modificar o que for necessário.

Além disso, com a adição do **PrototypeRegistro** (Registry), o sistema pode manter um catálogo de "templates" de Quizzes ou Trilhas frequentemente utilizados, permitindo ao administrador (`Client`) obtê-los e cloná-los facilmente.

## 4. Descrição das Classes

### 4.1. Interface `Prototype`

Interface base que define o contrato de clonagem.

| Método | Descrição |
| --- | --- |
| `+ clone(): Prototype` | Retorna uma cópia do próprio objeto. |

### 4.2. Classes Concretas (`Trilha`, `Modulo`, `Quizz`, `Questao`, `Alternativa`)

Classes do domínio que implementam o protótipo. A clonagem de uma classe "pai" (ex: Trilha) também deve clonar seus filhos (ex: Modulos) - o que caracteriza um _Deep Copy_ (Cópia Profunda).

| Classe | Atributos Principais | Métodos Principais |
| --- | --- | --- |
| **Trilha** | `id`, `titulo`, `descricao`, `ordem`, `qtdModulos` | `exibirModulos()`, `exibirProgresso()`, `marcarConcluido()`, `clone()` |
| **Modulo** | `id`, `titulo`, `descricao`, `ordem`, `qtdConteudos` | `exibirConteudos()`, `exibirProgresso()`, `marcarConcluido()`, `clone()` |
| **Quizz** | `id`, `titulo`, `qtdQuestoes` | `exibirProgresso()`, `iniciarQuiz()`, `clone()` |
| **Questao** | `id`, `enunciado`, `status`, `qtdAlternativas`, `ordem` | `validarResposta()`, `sortearOrdem()`, `clone()` |
| **Alternativa** | `id`, `descricao`, `alternativaCorreta`, `ordem` | `sortearOrdem()`, `clone()` |

## 5. Implementação

Abaixo, apresentamos a implementação do padrão Prototype aplicado a este contexto usando JavaScript. Os códigos demonstram uma classe modelo (`Trilha`), o Registro de Protótipos (`PrototypeRegistro`) e o arquivo principal de demonstração (`index.js`).

<details>
<summary><b>Ver Código Fonte</b></summary>

```javascript
// models/Trilha.js
const Prototype = require("../prototypes/Prototype");

class Trilha extends Prototype {
  constructor(id, titulo, descricao, ordem, qtdModulos) {
    super();
    this.id = id;
    this.titulo = titulo;
    this.descricao = descricao;
    this.ordem = ordem;
    this.qtdModulos = qtdModulos;
    this.concluido = false;
  }

  exibirModulos() {
    return `A trilha ${this.titulo} possui ${this.qtdModulos} módulos.`;
  }

  exibirProgresso() {
    return this.concluido ? "Trilha concluída." : "Trilha em andamento.";
  }

  marcarConcluido() {
    this.concluido = true;
  }

  clone() {
    return Object.assign(Object.create(Object.getPrototypeOf(this)), this);
  }
}

module.exports = Trilha;

// PrototypeRegistro.js
const Trilha = require("./models/Trilha");
const Modulo = require("./models/Modulo");
const Quizz = require("./models/Quizz");
const Questao = require("./models/Questao");
const Alternativa = require("./models/Alternativa");

class PrototypeRegistro {
  constructor() {
    this.prototipos = {
      trilha: new Trilha(1, "Engenharia de Requisitos", "Trilha base", 1, 3),
      modulo: new Modulo(1, "Introdução", "Módulo inicial", 1, 5),
      quizz: new Quizz(1, "Quiz Inicial", 4),
      questao: new Questao(1, "O que é requisito?", false, 4, 1),
      alternativa: new Alternativa(1, "Uma necessidade do usuário", true, 1),
    };
  }

  criar(tipo) {
    return this.prototipos[tipo].clone();
  }
}

module.exports = PrototypeRegistro;

// index.js (Demonstração de Uso)
const PrototypeRegistro = require("./PrototypeRegistro");

console.log("======================================");
console.log("  DEMONSTRAÇÃO — Padrão Prototype");
console.log("======================================\n");

const registro = new PrototypeRegistro();

const trilha = registro.criar("trilha");
console.log(trilha.exibirModulos());
console.log(trilha.exibirProgresso());
trilha.marcarConcluido();
console.log(trilha.exibirProgresso());

console.log();

const modulo = registro.criar("modulo");
console.log(modulo.exibirConteudos());

console.log();

const quizz = registro.criar("quizz");
console.log(quizz.exibirProgresso());
console.log(quizz.iniciarQuiz("yasmin@unb.br"));

console.log();

const questao = registro.criar("questao");
console.log(`Resposta correta? ${questao.validarResposta(1)}`);

console.log();

const alternativa = registro.criar("alternativa");
console.log(`Alternativa: ${alternativa.descricao}`);
```

## 6. Senso Crítico

A aplicação do **Prototype** na estrutura didática da plataforma provou ser extremamente vantajosa. A estrutura hierárquica e diversificada de `Trilha`, `Modulo`, `Quizz`, `Questao` e `Alternativa` significa que a criação manual de duplicatas de cursos ou módulos exigiria a instanciação repetitiva de cada um de seus componentes com estados semelhantes. Ao invés disso, utilizamos a estratégia de clonagem.

Na nossa implementação, utilizamos a abordagem nativa (_Shallow Copy_) através do uso conjunto de `Object.create` e `Object.assign`, que permite clonar o protótipo real das classes em JavaScript (incluindo métodos da herança) para o novo objeto. Essa escolha atende bem à replicação da base do objeto, evitando o acoplamento a construtores específicos por toda a aplicação e agilizando a duplicação dos dados simples de cada estrutura.

A introdução do `PrototypeRegistro` (Registry) atua como um repositório central de "templates", onde configurações comuns de questionários ou trilhas podem ser guardadas, recuperadas e clonadas facilmente. O cliente não precisa conhecer as particularidades de implementação de nenhuma das subclasses concretas — basta solicitar ao registro o tipo desejado ("trilha", "quizz", etc.), mantendo a responsabilidade única e ocultando a lógica de criação.

---

## 7. Conclusão

O padrão **Prototype** complementa perfeitamente as necessidades do gerenciamento de instâncias padronizadas na plataforma. Diferente dos padrões criacionais como Factory Method que focam em abstrair "qual" objeto é criado, o Prototype foca em abstrair "como" objetos já configurados são instanciados, baseando-se em cópias de estados anteriores de forma flexível e nativa à linguagem JavaScript. O seu uso para replicar o esqueleto de trilhas, módulos e quizzes traz clareza de código, reusabilidade e manutenibilidade para a arquitetura da plataforma.
