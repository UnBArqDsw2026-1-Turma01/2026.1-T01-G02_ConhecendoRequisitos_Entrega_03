## 1. Introdução

O **Facade** (Fachada) é um padrão de projeto estrutural que fornece uma interface simplificada (de alto nível) para um corpo complexo de código, como um subsistema, uma biblioteca ou um framework (GAMMA et al., 1995).

Em vez de tornar o código cliente diretamente dependente de dezenas de classes e lógicas de baixo nível, o padrão Facade atua como um "painel frontal" ou uma recepcionista. Ele esconde a complexidade e a orquestração dos componentes internos, expondo apenas os métodos que o cliente realmente se importa em utilizar.

## 2. Metodologia

Na plataforma **ConhecendoRequisitos**, já implementamos alguns padrões criacionais:
1. O **Builder** (`UsuarioDiretor`, `AlunoBuilder`, `AdministradorBuilder`) para lidar com a complexidade de instanciar perfis de usuários com parâmetros distintos.
2. O **Multiton** (`ConexaoDB`) para lidar com o *pool* de conexões de banco de dados isolados por contexto (`"usuarios"`, `"progresso"`, `"conteudo"`).

Contudo, ao desenhar os casos de uso reais como o fluxo de cadastrar usuário, percebemos que o Controlador/Front-end precisaria importar todos esses módulos, instanciar o Builder correto, pedir para o Diretor construir e, logo após, solicitar diferentes conexões ao Multiton para inserir os dados no banco.

Para evitar esse alto grau de acoplamento e respeitar os princípios arquiteturais, criamos a classe **`PlataformaFacade`**. Ela encapsula e coordena os padrões criacionais, permitindo que o cliente solicite ações complexas (como `cadastrarUsuario` e `concluirModulo`) através de uma única linha de código.

## 4. Implementação

O código a seguir apresenta a implementação da `PlataformaFacade` em **JavaScript (Node.js)**, coordenando os arquivos desenvolvidos nas seções anteriores.

<details>
<summary><b>Ver Código Fonte</b></summary>

```javascript
// 1. SUBSISTEMAS (Importando padrões criacionais)
const ConexaoDB = require("../../criacional/multiton/ConexaoDB");
const UsuarioDiretor = require("../../criacional/builder/UsuarioDiretor");
const AlunoBuilder = require("../../criacional/builder/builders/AlunoBuilder");
const AdministradorBuilder = require("../../criacional/builder/builders/AdministradorBuilder");

// 2. O PADRÃO FACADE 
class PlataformaFacade {
  constructor() {
    this.diretor = new UsuarioDiretor();
  }

  // O Facade decide sozinho qual Builder usar e quais Bancos (Multiton) chamar.

  cadastrarUsuario(tipo, nome, email, senha) {
    console.log(`\n[FACADE] Recebida solicitação para cadastrar um '${tipo}': ${nome}...`);
    let builder;
    // Escolhe o builder de acordo com o tipo solicitado
    if (tipo.toLowerCase() === "aluno") {
      builder = new AlunoBuilder();
    } else if (tipo.toLowerCase() === "administrador") {
      builder = new AdministradorBuilder();
    } else {
      throw new Error(`[ERRO] Tipo de usuário '${tipo}' não existe no projeto!`);
    }

    // 2. O Diretor monta o objeto complexo na memória (usando a ferramenta escolhida)
    const novoUsuario = this.diretor.construirUsuario(builder, nome, email, senha);

    // 3. Salva os dados padrão no banco global de usuários
    const dbUsuarios = ConexaoDB.getConexao("usuarios");
    dbUsuarios.query(`INSERT INTO Usuario (nome, email, tipo) VALUES ('${novoUsuario.nome}', '${novoUsuario.email}', '${novoUsuario.tipo}')`);

    // 4. Prepara o ecossistema específico para o tipo criado!
    if (novoUsuario.tipo === "aluno") {
      const dbProgresso = ConexaoDB.getConexao("progresso");
      dbProgresso.query(`INSERT INTO ProgressoTrilha (email_aluno, ofensiva) VALUES ('${novoUsuario.email}', ${novoUsuario.ofensivaAtual})`);
      console.log(`[FACADE] Ambiente educacional e banco de progresso preparados!`);
    } else {
      console.log(`[FACADE] Permissões concedidas. Ambiente administrativo pronto!`);
    }

    return novoUsuario;
  }

  concluirModulo(emailAluno, moduloId) {
    console.log(`\n[FACADE] Finalizando módulo ${moduloId} para: ${emailAluno}`);
    
    // Chama o banco A (Progresso)
    const dbProgresso = ConexaoDB.getConexao("progresso");
    dbProgresso.query(`UPDATE ProgressoModulo SET concluido = true WHERE email = '${emailAluno}'`);
    
    // Chama o banco B (Usuários)
    const dbUsuarios = ConexaoDB.getConexao("usuarios");
    dbUsuarios.query(`UPDATE Aluno SET ofensivaAtual = ofensivaAtual + 1 WHERE email = '${emailAluno}'`);
    
    console.log(`[FACADE] Módulo concluído e experiência (Ofensiva) atualizada com sucesso!`);
  }
}

// 3. TESTE PRÁTICO (Simulando o Cliente)
if (require.main === module) {
  const sistema = new PlataformaFacade();

  sistema.cadastrarUsuario("aluno", "yasmin", "yasmin@unb.br", "senha123");
  sistema.cadastrarUsuario("administrador", "arthur", "arthur@unb.br", "admin123");

  sistema.concluirModulo("yasmin@unb.br", 3);
}
```

## 5. Senso Crítico

Após fazermos os GoFs criacionais, a equipe identificou um desafio, onde a utilização dos padrões estava se tornando complexa para o cliente. Para cadastrar um usuário, o Front-end/Controlador precisaria compreender o Builder, instanciar o Diretor, solicitar as conexões adequadas ao Multiton e ainda gerenciar a lógica de inserção. O risco de gerar um alto acoplamento entre a camada de interface e a infraestrutura era grande.

Para solucionar esse problema, a adoção do Facade se mostrou ser uma ótima estratégia. Nós estruturamos essa classe para atuar como um orquestrador central do sistema. Agora, o Front-end apenas envia os dados básicos para cadastro, e o Facade assume a responsabilidade dos bastidores: ele seleciona o Builder apropriado, requisita as conexões aos bancos de usuários e de progresso, e realiza as inserções. Dessa forma, preservamos o Princípio da Responsabilidade Única (SRP) e simplificamos a utilização da nossa plataforma.

É importante ressaltar que, para manter a clareza desta documentação e focar no aprendizado do padrão, optamos por tratar a implementação do Facade como uma Prova de Conceito. Ao invés de implementar todas as funcionalidades previstas no nosso Backlog, nós focamos exclusivamente em `cadastrarUsuario` e `concluirModulo`. Essa escolha foi intencional, para demonstrar a eficiência do Facade: o primeiro método demonstra a orquestração do Builder com múltiplos bancos, enquanto o segundo lida com transações simultâneas de progresso.

Sobre o trabalho em equipe, a construção desta entrega foi um processo colaborativo, onde todos os integrantes participaram ativamente. A princípio, iriamos fazer o Bridge, mas percebemos que ele não se encaixava no nosso projeto e após algumas discussões, optamos por fazer o Facade. Foi muito legal perceber como os padrões de projeto das entregas anteriores se conectaram de forma tão coesa dentro do Facade, consolidando um orquestrador eficiente para o projeto. 


## 6. Conclusão

O padrão **Facade** provou ser uma ferramenta arquitetural elegante para ocultar a complexidade dos subsistemas. Ao unificar a comunicação com os padrões *Builder* e *Multiton*, garantimos que o código cliente interaja com o nosso domínio por meio de uma interface simplificada e com baixíssimo acoplamento. Esta camada de serviço protege as regras de negócio mais intrincadas de alterações na camada de apresentação, atestando a maturidade do design de software adotado pelo grupo.
